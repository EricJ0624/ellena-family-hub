# Naming Unification - Stage 1 Gate

목적: 기능명(앱 용어) 중심으로 파일/코드/API/DB 명칭을 점진 통일하되, 인증/가입/권한 회귀를 방지한다.

## 1) 작업 유형 선언

- 작업 유형: 로직/권한/연동 영향 가능성이 있는 리팩터링 작업
- 근거: DB 테이블명/쿼리 경로/API 참조가 연결되어 있어 연쇄 영향 가능

## 2) 영향 범위 선언

- 대상 범위
  - `app/features/**`
  - `app/modules/**`
  - `app/api/**`
  - `app/contexts/**`
  - `lib/**` (테이블 상수/쿼리 유틸/권한 보조 로직)
  - `docs/**` (네이밍 규칙/전환 기록)

- 변경 금지 영역 (명시적 승인 전)
  - 인증/가입/세션: `app/page.tsx`, `app/onboarding/**`, `auth` 관련 로직
  - 권한 판정/보안: `memberships`, `groups`, `profiles` 권한 분기
  - 라우팅 동작: 로그인/가입/초대 플로우 경로 및 분기
  - RLS/SQL 정책 변경, 인증 관련 API route 동작 변경

## 3) 승인 필요 여부 판정

- 사전 승인 필수
  - DB 마이그레이션(테이블 rename, 뷰/함수/트리거 변경)
  - RLS 정책/권한 로직 변경
  - 사용자 체감 동작(입력/버튼/에러/페이지 전환) 변경

- 승인 없이 가능한 범위
  - 명칭 문서화
  - 코드 내 하드코딩 테이블명을 상수로 치환(동작 동일 조건)
  - 폴더/파일명 정리 중 import 경로만 변하는 무기능 변경

## 4) 검증 계획 (고정 체크리스트)

- 정적 검증
  - `tsc` 통과
  - lint 통과

- 핵심 시나리오 검증 (매 배치 종료 시 반복)
  - 로그인 성공/실패 메시지와 라우팅 동일
  - 가입 후 안내/인증 콜백 동작 동일
  - 그룹 선택 및 권한별 화면 접근 동일
  - 가족 앨범/채팅/여행/첨부 업로드 기본 플로우 정상

- DB 영향 검증 (rename/SQL 포함 배치일 때)
  - FK/PK/인덱스 이상 없음
  - RLS 정책 정상 조회
  - 뷰/함수 의존성 깨짐 없음

## 5) 작은 단위 분할 계획

- 배치 원칙
  - 2~4파일 단위 또는 화면 1개 단위로만 변경
  - 한 배치에 DB 스키마 변경은 최대 1개
  - 배치 실패 시 즉시 롤백 가능해야 함

- 권장 순서
  1. 문서/상수화 배치 (무기능)
  2. 저위험 API 배치 (attachments 등)
  3. 기능별 배치 (family-chat, family-album, travel)
  4. 최종 DB rename 후보 배치 (사전 승인 + 롤백 SQL 필수)

## 6) 네이밍 기준 (초안)

- 앱/URL/기능 폴더: kebab-case (`family-chat`)
- DB 테이블/컬럼: snake_case (`family_chat_messages`)
- 코드 심볼: PascalCase/camelCase
- 참고: DB는 하이픈(`-`) 대신 snake_case 사용

## 7) 이번 단계 완료 기준

- 이 문서가 팀 기준으로 합의됨
- 이후 모든 배치는 본 문서의 게이트를 통과한 후에만 시작
- 승인 필요 항목은 구현 전 사용자 승인 기록을 남김
